## Problem Statement

Building a competitive Texas Hold'em poker AI requires deep integration of multiple complex systems: a game-theoretic solver (CFR-based GTO computation), real-time equity calculation, opponent behavior modeling, and a decision engine that can reason strategically under imperfect information. Traditional approaches (pure CFR solvers like Libratus/Pluribus, or end-to-end RL like AlphaHoldem) require massive computational resources and domain-specific engineering. Meanwhile, the user wants to leverage the reasoning capabilities of LLMs combined with precise mathematical tools to create a poker agent that thinks like a human expert — intuition and strategy from the LLM, backed by precise math from external tools — but without building all the agent infrastructure from scratch.

## Solution

**TexasAgent** is an LLM-powered Texas Hold'em poker AI agent that uses the pi coding agent as its reasoning runtime and tool-calling infrastructure, with a FastAPI web application providing the game engine, tool implementations, and interactive web UI.

The system has two sides:

**pi Package (`texas-agent`):** A pi package that registers poker-specific tools (equity calculator, GTO solver bridge, opponent tracker, range analyzer, hand strength evaluator) as pi extensions, plus a poker strategy knowledge base as pi skills. This package runs inside pi RPC mode, giving the LLM access to precise mathematical poker computations via function calling.

**FastAPI Web App:** A Python application that hosts the Texas Hold'em game engine (dealing, betting rounds, pot management, showdown), spawns pi in RPC mode as the agent decision engine, and provides a browser-based poker table UI with real-time updates via WebSocket.

### Core Flow

1. Player takes action in Web UI → FastAPI advances game state
2. When an AI agent needs to act → FastAPI sends game state to pi via RPC
3. pi's LLM agent reasons about the situation, calling registered poker tools as needed
4. pi returns a structured decision (fold/call/raise + amount + reasoning)
5. FastAPI applies the action, updates game state, pushes to all clients via WebSocket

### Why pi

- **Agent loop built-in:** Turn management, multi-round tool calling, error retry, streaming events
- **Tool registration:** `pi.registerTool()` with JSON Schema — poker tools become native LLM function calls
- **Skill system:** Poker strategy reference injected on-demand via `SKILL.md`
- **Session management:** Hand history, opponent reads, and agent memory persist across hands via pi's `SessionManager`
- **RPC mode:** Headless operation over stdin/stdout JSON — language-agnostic integration from Python
- **No reinvention:** Conversation compaction, model switching, thinking levels, error handling — all handled by pi

## User Stories

### Poker Gameplay

1. As a poker player, I want to play Texas Hold'em against AI opponents in a web browser, so that I can practice and study without needing other human players.
2. As a user, I want to see a visual poker table with community cards, player positions, chip stacks, and pot size, so that I can understand the game state at a glance.
3. As a user, I want to click buttons to fold, call, or raise (with a bet size slider/input), so that I can take actions naturally without typing commands.
4. As a user, I want to see AI opponents' decisions appear in real-time on the table, so that the gameplay feels responsive and immersive.
5. As a user, I want to see my hole cards only (not other players'), so that the game has proper hidden information.
6. As a user, I want to see a showdown animation that reveals all hands and declares the winner, so that I understand why the pot was awarded.
7. As a user, I want to configure the number of players (2–9 seats) and which seats are AI vs human, so that I can set up different game formats.
8. As a user, I want to configure starting stack sizes, blind levels, and whether it is cash game or tournament mode, so that I can simulate different poker variants.

### AI Agent Behavior

9. As a user playing against AI, I want the AI to make mathematically sound decisions informed by equity calculations, so that opponents are challenging and realistic.
10. As a user playing against AI, I want the AI to adapt its strategy based on my observed tendencies (VPIP, PFR, aggression frequency), so that it feels like playing against a thinking opponent rather than a static bot.
11. As a user, I want to see the AI's reasoning for each decision (in a collapsible panel or tooltip), so that I can learn from its thought process.
12. As a user, I want to configure different AI difficulty levels or playing styles (tight-aggressive, loose-passive, GTO-balanced), so that I can practice against various opponent types.
13. As a developer, I want the AI to call appropriate poker tools (equity, solver) only when needed — not every trivial decision — so that latency stays low and API costs stay reasonable.

### Hand Analysis & Learning

14. As a user, I want to review a completed hand with a full action-by-action replay, so that I can analyze my decisions after the fact.
15. As a user, I want to see equity percentages for each street of a reviewed hand, so that I understand how the odds shifted as community cards were revealed.
16. As a user, I want to see GTO solver output for key decision points in a reviewed hand, so that I can compare my play against theoretically optimal strategy.
17. As a user, I want to export a hand history as a standard format text file (PokerStars/HH format), so that I can import it into other analysis tools.

### Session & Statistics

18. As a user, I want to see cumulative session statistics (hands played, win/loss, BB/100, total profit), so that I can track my performance over time.
19. As a user, I want to see AI opponent statistics (VPIP, PFR, 3-bet%, c-bet%, aggression factor) that build up over multiple hands, so that I can study their tendencies.
20. As a user, I want my session to persist across browser refreshes and server restarts, so that I can resume a long session.
21. As a user, I want to start a new session or load a previous session, so that I can manage multiple play sessions.

### Administration & Configuration

22. As a system operator, I want to configure which LLM provider and model each AI seat uses (Claude, GPT, etc.), so that I can experiment with different reasoning backends.
23. As a system operator, I want to configure the path to the TexasSolver binary or disable GTO solving (equity-only mode), so that the system can run without optional heavy dependencies.
24. As a developer, I want to run the poker engine in a headless simulation mode (no Web UI) to batch-test AI decision quality across thousands of hands, so that I can evaluate agent performance.

## Implementation Decisions

### System Architecture: Dual-Process

The system runs as two communicating processes: a Python FastAPI server (game engine + web UI) and a Node.js pi RPC subprocess (LLM agent + poker tools). They communicate via JSON-over-stdin/stdout using pi's RPC protocol.

**Decision:** pi runs in `--mode rpc --no-session` mode. FastAPI spawns and manages the pi process lifecycle. Game state is formatted as a structured text prompt; the LLM responds with a structured action parsed from its tool calls and final message.

### pi Package: `texas-agent`

The pi package registers six custom tools as pi extensions:

| Tool | Input | Output | Implementation |
|------|-------|--------|----------------|
| `poker_equity` | hole cards, community cards, opponent count | win/tie/lose percentages | Python subprocess (phevaluator Monte Carlo) |
| `poker_solve` | board, pot, effective stack, hero range, villain range | GTO strategy (action frequencies per combo) | TexasSolver console_solver subprocess |
| `poker_hand_strength` | hole cards, community cards | hand rank description + percentile | Inline TypeScript (static lookup) |
| `poker_pot_odds` | pot size, call amount, implied odds estimate | pot odds ratio, required equity % | Inline TypeScript (pure math) |
| `poker_opponent_stats` | player ID | VPIP, PFR, 3bet%, cbet%, AF, WTSD, hand count | SQLite query |
| `poker_range_analysis` | range string, board | equity distribution heat, range advantage | Python subprocess (equity.py --range mode) |

The package also includes a `poker-strategy` skill (`SKILL.md`) containing GTO fundamentals, position-based opening ranges, bet sizing theory, and opponent exploitation principles. This skill is injected into the LLM's system prompt by pi's skill system.

### FastAPI Poker Engine

The poker engine is a pure-Python, deterministic state machine implementing No-Limit Texas Hold'em rules:

- Deck management (shuffled, deterministic seeded for testing)
- Blinds posting, dealing (2 hole cards per player, 5 community cards: flop/turn/river)
- Betting rounds with legal action enumeration (fold, check, call, bet/raise min to all-in)
- Pot management including side pots for all-in scenarios
- Showdown hand evaluation (best 5-card hand from 7 cards) using phevaluator or Pokerkit
- Winner determination and pot distribution

**State machine shape:**

```
GameState {
  players: Player[]           // seats, stacks, hole cards, folded status
  community_cards: Card[]     // 0, 3, 4, or 5 cards
  pot: number                 // main pot
  side_pots: SidePot[]        // for all-in scenarios
  current_bet: number         // current bet to match
  dealer_index: number        // button position
  acting_player_index: number // whose turn
  round: "preflop" | "flop" | "turn" | "river" | "showdown"
  min_raise: number           // minimum raise size
  last_aggressor_index: number
}
```

### Decision Protocol: FastAPI ↔ pi

FastAPI sends game state to pi as a structured natural language prompt. pi's LLM agent calls tools, reasons, and produces a final action. FastAPI parses the tool-result messages from pi's event stream to extract structured data.

The prompt includes: hole cards, community cards, pot size, stack sizes, position, action to call, opponent statistics, and previous actions this hand. A reserved tool `poker_decide` may be used to force a structured final decision output.

### Web Frontend

Server-rendered HTML with HTMX for partial updates and a WebSocket connection for real-time game state push. No SPA framework required — the FastAPI server manages all rendering.

**Technology:** Jinja2 templates + HTMX + native WebSocket (JavaScript). This keeps the entire stack Python-centric for the web layer, with pi handling only the LLM agent concerns.

### Database Schema

SQLite database with tables:
- `sessions` — session metadata (start time, end time, table config)
- `hands` — per-hand records (hand number, community cards, pot size, winner, net result per player)
- `actions` — per-action records (hand ID, player, street, action type, amount)
- `player_stats` — aggregated opponent statistics (VPIP, PFR, 3bet, cbet, AF, WTSD, hand count) updated incrementally

### System Prompt & Agent Memory

The system prompt for the LLM agent includes poker rules summary, strategic principles from the poker-strategy skill, tool usage instructions, and decision output format specification.

Agent memory across hands is managed by pi's `SessionManager`. Each hand is a new turn in the pi session. Opponent observations are stored as custom entries appended to the session. On each decision, the LLM has access to running opponent stats via `poker_opponent_stats` tool calls.

## Testing Decisions

### What makes a good test

Tests verify external behavior, not implementation details. For the poker engine: given a game state and action, assert the resulting state matches poker rules. For the agent: given a game state, assert the returned action is legal and reasonable (within a tolerance for LLM non-determinism). For tools: given inputs, assert correct outputs.

### Seam 1: Agent decision function (highest integration seam)

`decide(game_state) → Action` — the boundary between FastAPI and pi RPC. This is the ideal single seam covering the core behavior chain.

**Test approach:** Integration tests that send known game states to pi, verify returned actions are legal. Use deterministic deck seeds for reproducibility. Test categories: preflop decisions (open raise, 3-bet, fold to 4-bet), postflop decisions (c-bet, check-raise, bluff catch), river decisions (value bet sizing, bluff frequency).

### Seam 2: Poker engine (deterministic unit tests)

The engine is a pure function: `apply_action(state, action) → new_state`. Every poker rule is testable in isolation without LLM involvement.

**Test approach:** Unit tests for dealing (correct number of cards, no duplicates), betting (legal action enumeration, min-raise calculation, all-in side pots), hand evaluation (correct winner determination for edge cases like split pots, kicker resolution).

### Seam 3: Tool layer (deterministic unit tests)

Each tool accepts structured input and returns structured output. No side effects (except opponent_stats which reads SQLite).

**Test approach:** Unit tests with known inputs. Equity calculator: known equity values for AA vs 72o (~85%). Solver: basic spots where GTO strategy is known (e.g., BTN vs BB single-raised pot).

### Seam 4: WebSocket protocol (API contract tests)

**Test approach:** FastAPI TestClient with WebSocket. Verify game state messages match schema, actions are correctly propagated to all connected clients.

### Prior art

This is a greenfield project. Test patterns follow standard pytest conventions: fixture-based setup, parametrized test cases, deterministic seeding for reproducibility. The poker engine tests follow the pattern of board game engine testing (given state + action → assert resulting state).

## Out of Scope

- **Real-money play or gambling integration.** This is purely a research/educational tool.
- **Online poker site integration or screen scraping.** The agent only plays within its own hosted game.
- **Training the LLM or fine-tuning models.** The LLM operates zero-shot/few-shot via prompting and tool use.
- **Multi-table support.** v1 is single-table only.
- **Mobile app.** Web-only for v1.
- **Omaha or other poker variants.** Texas Hold'em only.
- **Full CFR solver implementation in this repo.** We wrap existing solvers (TexasSolver) rather than implementing CFR from scratch.
- **Authentication or user accounts.** Single local user for v1.
- **Deployment to production servers.** Local development and single-machine usage only for v1.

## Further Notes

### Key References

- **ToolPoker (ICLR 2026, Microsoft Research):** Demonstrated that LLMs with external solver tools can approach Nash equilibrium in Limit Hold'em. Our architecture follows the same pattern but for No-Limit.
- **PokerSkill (2025):** Showed that structured prompts can dramatically improve LLM poker performance even without solver queries. We combine both: structured prompts + solver access.
- **mcp-poker (alanyangrice):** Existing MCP server with 14 poker tools. We can draw inspiration from its opponent tracking and equity calculation patterns, though we implement tools as pi extensions rather than MCP.
- **openpoker (philippe-page):** Python poker engine with LLM agent support. Reference for engine architecture and context assembly patterns.
- **TexasSolver (bupticybee):** Open-source C++ GTO solver we wrap for GTO strategy computation.
- **PokerKit / phevaluator:** Python libraries for hand evaluation and equity calculation.

### Risk: LLM Non-Determinism

LLMs produce different outputs for the same input. Mitigation: constrain the action space through structured output (function calling with legal action enum), validate all returned actions against game rules before executing, and use temperature=0 for critical decision points. For test assertions on agent behavior, use statistical assertions over many hands rather than exact matches.

### Risk: Latency

LLM inference + tool calls can take 2–10 seconds per decision. Mitigation: cache equity and solver results for repeated spots, use cheap models (Claude Haiku) for trivial decisions (folding 72o preflop), pre-compute common preflop ranges, show "thinking" indicator in UI during agent deliberation.

### Risk: Tool Call Budget

LLMs may call tools excessively, burning API costs. Mitigation: cap tool calls per decision (max 3), design the system prompt to prefer equity over solver for most spots, make opponent stats readily available in the game state prompt to reduce tool calls.
